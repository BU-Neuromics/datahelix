# BASS-Hippo

The BASS-Hippo component is a standalone data storage engine that can be used independently of other BASS components. It provides a REST API for data storage and retrieval, and can be used with other BASS components or third-party storage systems.

## Installation

See the [BASS-Hippo Installation Guide](installation.md) for instructions on how to install and configure BASS-Hippo.

## Data model

Hippo implements a composable data model that captures all the key data types and relationships require for the storage, querying, analysis, and sharing of biological data. The data model is subdivided into modules that can be composed independently of each other depending on the needs of the user. This allows BASS deployments to have the minimum set of data storage infrastructure required for the user's specific use case.

The core data model is required for all BASS deployments. This model includes the following data types:

- Deployment information
- System configuration settings
- System users
- System groups
- Authentication and authorization information
- User and Group permissions and roles
- Data classification and security levels
- API access and rate limiting information
- Basic logging and monitoring data
- Global metadata schemas
- Version control information
- BASS system interconnection information

The following optional modules are currently supported:

- **Speciment Tracking** - for biological sample tracking
- **Subject Tracking** - for tracking human subjects
- **Experiment Management** - for tracking experimental protocols, conditions, and results
- **Inventory Management** - for managing lab supplies, reagents, and equipment
- **Data Store** - for tracking data files, metadata, and provenance
- **Workflows** - for tracking and executing workflows
- **Collaboration** - for managing shared projects, team permissions, and external collaborations
- **Honest Broker** - for data de-identification, anonymization, and re-identification
- **Audit Trail** - for comprehensive logging of all data modifications and access
- **Analytics** - for storing statistical analyses, visualizations, and derived data


Some optional modules depend on others for deployment. For example, the **Experiment Management** module requires the **Speciment Tracking** module to be deployed. Others can optionally be delpoyed and interface with each other via overlays. For example, the **Workflows** module can interface with the **Data Store** module to track the files and metadata generated by the workflows.
