    Gen3 — Deep Comparison to BASS / Aperture¶

    Status: 🟢 Research findings (2026-06-16, primary sources). Companion to prior-art.md, which named Gen3 the strongest faceted-portal match. This goes deeper: Gen3 is the closest production analog to the whole BASS platform, not just to Aperture — and its architecture answers our sharpest open capability question.

    What Gen3 is¶

    Gen3 (Center for Translational Data Science, U. Chicago / uc-cdis) is an open-source platform for building data commons — it powers NIH commons like MIDRC, AnVIL, BioData Catalyst, HEAL. It is a stack of microservices around a config-driven data model (the Data Dictionary) and a config-driven faceted Data Explorer. Many independent commons run the same codebase differentiated by dictionary + portal config — the same "generic engine, per-instance config" thesis as BASS/Aperture.

    Service-by-service map to BASS¶

    Gen3 service Role BASS analog
    Data Dictionary (YAML, one JSON-Schema file per node; nodes + typed properties + links w/ backref, multiplicity, semantic label; Program = graph root) Config-driven data model Hippo LinkML schema (entities + slots + relationships). LinkML is more expressive/standardized than Gen3's per-node JSON-Schema.
    Sheepdog Submission/write into the Postgres graph; validates against the dictionary (rejects INVALID_LINK) Hippo write path + validation
    Peregrine GraphQL over the normalized Postgres graph (traversal, lineage) Hippo's autogenerated GraphQL (graph-shaped, normalized)
    Tube (ETL) Flattens the graph → Elasticsearch documents; re-runs on new submissions (no Hippo equivalent today)
    Guppy GraphQL over Elasticsearch — faceted search, aggregations, counts, histograms, tiered access (no Hippo equivalent — this is the gap, see below)
    Indexd GUID → physical file location index Hippo "where data lives" / Canon
    Fence AuthN — OIDC/OAuth identity provider; issues tokens carrying arborist policies Bridge (authn half)
    Arborist AuthZ — RBAC policy engine; fence embeds its policies in tokens Bridge (PEP/PDP half)
    Data-portal (Windmill) React frontend; the Data Explorer is driven by gitops.json/explorerConfig (config-as-data) Aperture
    Hatchery / MDS / Pelican / Sower / WTS Workspaces, metadata svc, PFB export, batch jobs, tokens (various: Cappella / platform)

    fence + arborist = Bridge. Gen3 independently arrived at exactly Bridge's split: a central OIDC identity provider plus a separate policy engine whose decisions ride in the token. Strong external validation of the Bridge PEP/PDP design (ADR-0016, platform sec6).

    The explorerConfig ≈ our core-loop config (near 1:1)¶

    Gen3's gitops.json explorerConfig is config-as-data that drives the faceted explorer — strikingly close to what our prefab core-loop config will look like:

    explorerConfig block Concrete shape Maps to
    filters.tabs[].fields[] facet fields grouped into titled tabs PF-3 facet panel (grouped facets)
    charts{field: {chartType: count\|pie\|fullPie\|bar, title}} charts as data, keyed by field PF-6 charts bound to query
    table{enabled, fields[], linkFields[]} table columns as data PF-2 collection table
    buttons[]{type: data-csv\|manifest\|export-to-pfb, title, fileName} export actions as data PF-7 export
    guppyConfig{dataType, fieldMapping[], manifestMapping{...}} binds the tab to a Guppy ES index + field renames + cohort→file join binding (which collection; relationship/file join)

    And the validation discipline matches our composition+binding split: only properties present in etlMapping.yaml may appear in gitops.json, and it's validated against the Data Dictionary + ETL mapping (via gen3utils CI). This is config-as-data with schema-derived binding — proven across many commons.

    The single biggest lesson: a two-tier data architecture solves our aggregation gap¶

    The capability gaps we hit in core-loop Steps 2–3 (no sort, no counts, no aggregation/facet counts over Hippo's thin GraphQL) are exactly what Gen3 solved — by running two query surfaces over two stores:

    • Peregrine / Postgres graph — normalized; great for traversal, lineage, submission validation. (≈ Hippo's GraphQL today.)
    • Guppy / Elasticsearch — a denormalized, indexed, aggregation-optimized copy, kept in sync by the Tube ETL. Purpose-built for fast faceted search, counts, and histograms.

    Hippo's current GraphQL is the Peregrine analog. Aperture's faceted explorer needs the Guppy analog — and Hippo has none yet. This is the decisive architectural question for the core loop: - (b) server-side index layer: give Hippo (or a sidecar) a Guppy-equivalent — ETL/flatten to an aggregation store (ES / DuckDB / OLAP) with a faceting+aggregation GraphQL surface. This is Gen3's production answer. - (c) adapter compensation: Aperture's Layer-D adapter computes aggregation client-side (DuckDB-WASM) when the source can't — viable at small scale, breaks on large cohorts.

    Gen3's experience says facet counts at real scale want the index layer (b). This likely warrants a Hippo design issue.

    Also worth stealing: Guppy's tiered access (private / regular / libre) lets users see aggregate counts with a minimum-count threshold over data they cannot access row-level — a real biomedical-privacy pattern for Bridge's field-mask/predicate model (aggregate-with-min-count prevents re-identification).

    Where Aperture is more ambitious than Gen3¶

    Dimension Gen3 Aperture
    Generic over backends Its own stack only Any LinkML+GraphQL endpoint
    Config form JSON in a gitops repo, CI-validated, portal redeploy Serializable LinkML data in the backend, structurally validated, versioned/provenance-tracked
    New visualization / behavior Code a React component in data-portal (bespoke) Typed sandboxed component emitting a view-description, hot-reloaded (ADR-0009/0011)
    Authoring Humans edit gitops.json + redeploy Humans and LLM agents edit config (MCP/API, dry-run, ADR-0018/0021)
    View vocabulary Fixed chart set baked into Windmill Growable typed noun-catalog (Vega-Lite-style, ADR-0010)

    What to steal / prototype against¶

    1. The explorerConfig schema as the concrete template for our core-loop config (facets-as-tabs, charts-by-field, table fields, export buttons, dataType binding).
    2. The validation model (gitops.json validated against dictionary + ETL mapping) — but make ours structural/live (LinkML-in-Hippo) rather than CI-time tooling.
    3. The two-tier graph+index architecture as the reference answer to faceting/aggregation at scale (the Guppy/Tube pattern).
    4. manifestMapping (cohort → file-manifest join) for PF-7 export.
    5. Tiered aggregate access + min-count for Bridge's privacy model.

    Caveats¶

    • Gen3 is domain-specific (NIH biomedical data commons) and heavyweight (a dozen+ microservices); not adoptable wholesale.
    • Its portal is bespoke React (Windmill) — not generic, not sandboxed, not agent-editable. The config drives a hand-built app; it is not the generic config-driven engine we're after.
    • Refines prior-art.md's shorthand ("not an introspectable GraphQL transport"): Gen3 does use GraphQL (Peregrine over the graph, Guppy over ES), but its portal binding is config-mapped to ES indices, not auto-derived from schema introspection — so the derived-binding bet remains ours to prove.
