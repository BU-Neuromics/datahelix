    Prefab Portal — The Core Loop (PF-2 browse ⇄ PF-3 filter)¶

    Status: 🟠 Walkthrough in progress. We build this step by step; each step records UX + concise implementation approach + open questions (tagged (a) supported / (b) Hippo enhancement / (c) adapter compensation — see README capability theme). Grounded in the omics/brain-bank schema.

    What the core loop is¶

    The exploration cycle every data portal lives or dies by:

      ┌──────────────────────────────────────────────────────────┐
  │  pick a collection → see a table → refine (facets/search) │
  │        ↑                                    │              │
  │        └──────── table updates ◄────────────┘             │
  │                         │                                 │
  │              drill into a record / pivot / chart / save   │
  └──────────────────────────────────────────────────────────┘

    Step map¶

    1. Entry / collection selection — nav of collections; land on one.
    2. The collection table — columns, rows, sort, pagination, count, states.
    3. The facet + search panel — facet types, counts, free-text, filter semantics. (next)
    4. Query state ⇄ URL ⇄ results — the serializable query-state object; live refinement; shareable.
    5. Exit / pivot — row → detail (PF-4); relationship pivot (PF-5); feed charts (PF-6) / save (PF-7).

    Step 1 — Entry / collection selection¶

    UX¶

    • A nav (left sidebar) lists the collections the portal exposes — for the brain-bank prefab: Donors (Subject), Samples, Brain Samples, Datafiles, Datasets, Workflows. The designer hand-picks which appear, their order, label, and icon.
    • Selecting a collection opens its collection view = table (Step 2) + facet panel (Step 3).
    • A default collection (or the PF-1 dashboard) is the landing.
    • Display label ≠ schema class: the nav shows "Donors," the schema class is Subject. The label lives in (later) config; the class name stays in data.

    Implementation approach¶

    • Collections derive from GraphQL introspection: Hippo's hippoSchema lists exposed entity types (richer than standard __schema — it also gives slot kinds, enum values, relationship targets). Generic baseline can fall back to __schema.
    • Prefab: a hand-authored nav list [{type, label, icon, order}]. This is the Level-1 composition seed — but in the prefab phase it is just hand-built, not yet config.
    • is-a inheritance (BrainSample → Sample): a "Brain Samples" collection is a filtered/typed view over Sample, or its own collection — open question (see below).

    Open questions¶

    • Q1.1 Is BrainSample its own collection, or a saved filter over Samples? (Polymorphic is-a — queryable as Sample.) (design)
    • Q1.2 Does the nav list all exposed types by default (generic baseline) with the prefab hiding/reordering, or is it an explicit allow-list? Leaning: derive-all + override (DC-2/3).

    Step 2 — The collection table¶

    UX¶

    • Header row of columns; data rows; each row links to the entity detail (PF-4).
    • Curated column subset, not every slot. Donors (Subject) default columns: external_id · species · biological_sex · age_at_collection · diagnosis · #samples.
    • Sortable column headers (where supported).
    • Pagination (page size + next/prev or numbered) and a total count ("1,243 donors").
    • States: loading skeleton · empty ("no matching donors") · error.
    • Cell rendering keyed by slot type: enum → label/chip · float → formatted number · date → formatted · uri → link · relationship → count or chips.

    Implementation approach¶

    • One GraphQL list query per collection, e.g. subjects(filter: {...}, limit: 25, offset: 0) { id externalId species biologicalSex ageAtCollection diagnosis }. Hippo's autogenerated list gives equality filters + AND/OR + offset pagination (a).
    • The table is itself the first catalog primitive (ADR-0010 seed): a collection-table parameterized by {columns: [{slot, renderer}], sort, page, filter}. Cell renderers are a small typed registry keyed by LinkML slot kind. Hold this thought for DC-4.
    • Column set = composition + binding config later; in the prefab, hand-authored per collection.

    Open questions / capability gaps¶

    • Q2.1 Sort — Hippo's GraphQL list advertises filters + pagination + FTS but not order_by. → (b) propose a Hippo order_by capability; until then (c) client-sort the current page only, clearly scoped, or disable sort. Track as Hippo enhancement.
    • Q2.2 Total count — "1,243 donors" needs a count; no totalCount/count aggregate is advertised. → (b) propose totalCount on the list connection; interim (c) show "25+ / page N" without a grand total.
    • Q2.3 Derived relationship column (#samples) — a per-row related count. Hippo resolves relationship fields via DataLoaders, but a count may not be a field. → (b) aggregate/ count capability, or (c) omit derived counts in the v1 baseline.
    • Q2.4 Multivalued slots return [] in the v0.1 SQLite adapter — relationship/chip columns may render empty; degrade gracefully (treat absent as "not available," per ADR-0016).
    • Q2.5 isAvailable is always true on reads today — availability columns/filters are moot for now.

    Step 3 — The facet + search panel (next session step)¶

    To be developed: facet types per slot annotation (enum→checklist+counts, numeric→range, date→range, fts→search box), facet counts (the big aggregation question — likely the sharpest capability gap), AND/OR + range semantics, active-filter chips, clear-all.

    Step 4 — Query state ⇄ URL ⇄ results (later)¶

    The serializable query-state object — current collection + filters + sort + page — that drives the query, binds to the URL (shareable), feeds charts (PF-6), and is the precursor to the saved-view (PF-7) and to view-description-as-data (ADR-0009/0010).

    Step 5 — Exit / pivot (later)¶

    Row → detail (PF-4); facet/click on a relationship → pivot to the related collection (PF-5); current query → chart (PF-6) or save (PF-7).
