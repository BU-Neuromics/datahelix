# Comprehensive Test Suite Implementation

## Goal
Comprehensive Test Suite Implementation: Implement comprehensive test suite with unit tests per module and integration tests covering REUSE path, BUILD path with dependency resolution, and cycle detection.

## Acceptance Criteria
- Given the test suite is run with `pytest --cov=canon --cov-report=term-missing` and cli/ entry points are excluded via `--cov-config` or `--ignore`, when coverage is aggregated across all canon/ subpackages (core_types, rules_dsl, entity_ref_resolver, recursive_planner, cwl_executor, output_ingestion), then line coverage for each individual subpackage is at least 80% and the overall canon/ package (excluding cli/) achieves at least 80% line coverage
- Given a `tests/unit/` directory exists with one test file per canon module following the naming convention `test_<module_name>.py`, when `pytest tests/unit/ -v` is run, then every test file is collected and all tests pass with exit code 0
- Given each unit test module in `tests/unit/`, when the test functions are counted per module, then each module contains at least 3 test functions explicitly covering a normal/happy-path scenario, an error/exception-path scenario, and an edge-case scenario (e.g., empty input, boundary values, or None arguments)
- Given a mock Hippo server fixture is configured to return a pre-existing `AlignmentFile` entity with URI `hippo://alignments/sample-001` for a known spec lookup, when the REUSE integration test invokes `canon get` with that spec, then the returned result contains the entity URI `hippo://alignments/sample-001`, no CWL executor adapter method is called (assert mock call count is 0), and the test completes in under 5 seconds
- Given a mock Hippo server fixture configured to return no matching entity for a `trim_reads` rule target and a `canon_rules.yaml` containing a `trim_reads` rule with defined inputs and CWL tool reference, when the BUILD integration test invokes the full resolve-and-execute pipeline, then `CwltoolAdapter.run` is called exactly once with an `inputs.json` dict matching the rule's declared input bindings, and the mock Hippo server's `create_entity` method is called once with the output entity type and attributes specified in the rule's output mapping
- Given a `canon_rules.yaml` containing two rules `rule_a` and `rule_b` where `rule_a` depends on the output of `rule_b` and `rule_b` depends on the output of `rule_a`, when the cycle detection integration test calls `RecursivePlanner.resolve()` for either rule, then a `CanonCycleError` is raised whose message contains the cycle path as an ordered list (e.g., `['rule_a', 'rule_b', 'rule_a']`), and no CWL executor method is invoked prior to the error
- Given a `canon_rules.yaml` containing a single rule with a self-referential dependency (rule depends on its own output), when `RecursivePlanner.resolve()` is called for that rule, then a `CanonCycleError` is raised identifying the single-node cycle
- Given a three-level dependency chain in `canon_rules.yaml` (rule_c depends on rule_b, rule_b depends on rule_a, rule_a has no dependencies) and mock Hippo returns no pre-existing entities, when the BUILD integration test resolves rule_c, then the CWL executor is invoked exactly 3 times in dependency order (rule_a first, then rule_b, then rule_c) and all 3 output entities are ingested into mock Hippo
- Given the full test suite (unit and integration tests), when `pytest tests/ --tb=short` is run in a clean virtual environment with only the declared project dependencies installed, then all tests pass without requiring external services, network access, or filesystem side effects outside of `tmp_path` fixtures

## Constraints
- Complexity: high
